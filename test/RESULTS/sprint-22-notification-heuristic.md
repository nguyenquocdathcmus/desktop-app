# Sprint 22 US-174 — Notification Auto-Detect Heuristic Measurement

## What was measured

`NotificationDetector` (`src/main/notification-detect/NotificationDetector.ts`) samples 1
grayscale frame/second at 64×36 via the real bundled ffmpeg binary, diffs an 8×8 cell grid
between consecutive frames, and looks for cells in the top-right quadrant that jump sharply
(appear), stay stable, then jump sharply again (disappear) within a 2-7s window — the
documented macOS notification banner signature from `sprint/SPRINT_19.md`.

Automated measurement: `test/integration/notification-detector.test.ts`, run via
`npm run test:integration`. Fixtures generated by `scripts/generate-notification-fixtures.sh`
(regenerates automatically if missing).

## What this does and doesn't prove

The synthetic fixtures deliberately reproduce the exact signature described in the Sprint 19
spec (corner position, appear/hold/disappear timing, contrast against background) — passing
them proves the state-machine logic is implemented correctly, not that it generalizes to real
notification banners with their actual visual characteristics (frosted-glass translucency,
app icon, animated slide-in, text content, real macOS window shadow).

Two fixture-generation bugs were caught and fixed during this measurement, which is exactly
why measuring against real data matters more than eyeballing the logic:
1. The first fixture revision used a banner color (`#1c1c1e`) too close to the background
   (`#2c2c2e`) — luma delta peaked at 0.04, under any reasonable detection threshold. This
   would have been invisible without actually running ffmpeg and inspecting the diff values.
2. `libx264 -crf 20` softened the drawbox edges enough to blunt the corner-cell diff further;
   dropping to `-crf 18` restored a sharp enough edge for the grid-cell diff to fire cleanly.

## Real footage (US-173) — not yet done, and why it's still the honest gate

Sprint 19 flagged real notification footage as the actual blocker, and that remains true here:
this environment has no way to trigger a real macOS notification banner deterministically
inside a test run (no scriptable way to fire Slack/Mail/Calendar notifications on demand from
a sandboxed CI-style process). Manual procedure for a developer machine, to run before
enabling this in the product UI:

1. Record a 2-3 minute screen session with Slack, Mail, and Calendar notifications enabled,
   deliberately triggering 5-10 real notifications during the recording (message a colleague,
   send a test calendar invite for "now").
2. Run `NotificationDetector.detect()` against the resulting file, note timestamps by hand
   from scrubbing the video, and compare recall/false-positive rate to the table above.
3. If real-footage false-positive rate exceeds 5%, do **not** ship US-154/US-158 into the UI —
   tighten `CHANGE_THRESHOLD`/duration window in `NotificationDetector.ts` and re-measure, or
   keep this Blocked another sprint. Do not lower the ship bar to match a weaker result.

**Current status: synthetic measurement passed; real-footage measurement (step above) still
needs a developer to run it once on real hardware with real notifications before US-154 is
wired into `RecordingControls`/Timeline UI.** The detector module and its test harness are
complete and committed so that step is now a single manual verification, not a development task.
